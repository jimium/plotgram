1. 执行摘要

核心结论：不新增 diagram c4 类型。C4 是 architecture 上的分层语义 + 视图投影模式； 手写/Agent 走 DSL 属性；企业规模化走 Compose 多视图 API。差异化在「一份模型、多视图、可 Diff」而非复刻 Structurizr 全功能。
决策项选择理由
图表类型 扩展 architecture entity / relation / group 与 C4 组件模型同构；复用 architecture-v2 布局与正交路由
模型与视图 单一 AST + view 投影 对齐 Structurizr「一模型多视图」；避免 Mermaid 每种视图重写 DSL
包含关系 group 嵌套 + level 标注 Software System / Container 用 group 边界；Component 为组内 entity
企业出图 Connector → Compose views K8s Namespace→Container 聚合与 C4 Container 视图天然契合（见 enterprise 战略文档）
第一交付 Context + Container 视图 覆盖 80% 架构评审与 AI 对话场景；Component 次之；Code / Deployment 后置

2.3 用户真正买的是「分层沟通」，不是第四层代码图

调研 Structurizr 官方文档与企业实践：绝大多数团队停留在 Context + Container， Component 用于核心子系统，Code 层几乎只出现在 IDE 反向工程演示中。 Drawify 应将工程资源集中在视图投影、关系聚合、Diff、Connector，而非类图渲染。

3.1 竞品启示

  • 向 Structurizr 学：单一模型、多视图、层级标识符；企业 Landscape 用 workspace 扩展合并。
  • 避开 Structurizr 的复杂度：不引入完整 workspace DSL；用 Drawify 已有 group + Compose 达到 80% 效果。
  • 碾压 Mermaid 的点：一份 DSL 切换视图、全局 entity ID、validate 可修复、Diff 可审查。
  • 不对标 PlantUML：不在 Code 层与 UML 类图竞争。

5. 设计原则

  1. C4 是视图模式，不是新图表物种 — 避免 DiagramType::C4 膨胀。
  2. 语法增量极小 — 仅增加标准属性 level、diagram 属性 view / scope；符合「为 AI 设计」。
  3. 单一事实来源 — 全量模型存一份 AST；视图是投影，不复制 entity。
  4. group 表达边界 — Software System 与 Container 优先用 group；减少「扁平 entity + level」歧义。
  5. 确定性 — 投影、聚合、过滤均按 id 排序（遵循 AGENTS.md §2）。
  6. 手写与自动同源 — Connector 写的 JSON AST 与 Agent 写的 DSL 解析后结构一致。
  7. 渐进交付 — 无 view 时行为与今日 architecture 完全相同。

deployment 与 container 正交，仅在 deployment 视图激活

7. 核心方案：单一模型 + 多视图投影

7.1 双入口，同一语义

入口 A — 手写 / Agent DSL

架构师或 LLM 编写完整 architecture 模型，通过 view 选择投影。

  • 适合 PR 小改、对话出图
  • diff2 直接比较完整模型

入口 B — Connector + Compose

从 K8s / CMDB 构建 JSON AST，Compose 规则标注 level 并生成多视图请求。

  • 适合规模化、定时同步
  • 同一 snapshot-id 多 view 输出

7.2 diagram 级视图属性

属性合法值含义
view context | container | component | deployment | landscape 激活 C4 投影；缺省则不投影
scope group / entity id component 视图聚焦某一 Container（group）
theme builtin.c4 推荐与 C4 视图联用

7.3 entity / group 级属性

属性挂载合法值说明
level entity, group context | container | component | code | deployment C4 层级标注；参与投影过滤
description entity, group 字符串 对应 C4 副标题（技术栈说明）
tags entity 字符串列表 过滤视图、样式分支（可选 P2）
drilldown entity, group（可手写；通常由引擎注入） 对象 { view, scope?, title? } 视图间导航元数据，见 §18.3

7.4 完整示例（单文件多视图模型）

diagram architecture {
    title: "Internet Banking System"
    view: container
    theme: builtin.c4

    // ── Context 层实体 ──
    entity customer "Personal Banking Customer" {
        type: frontend
        level: context
        description: "A customer of the bank, with personal banking accounts"
    }
    entity email "E-mail System" {
        type: external
        level: context
        description: "Internal Microsoft Exchange system"
    }

    // ── 目标系统（Software System）──
    group banking "Internet Banking System" {
        level: context
        description: "Allows customers to view accounts and make payments"

        entity web "Web Application" {
            type: frontend
            level: container
            description: "Delivers the static content and SPA"
        }
        entity app "API Application" {
            type: service
            level: container
            description: "Provides Internet banking functionality via API"
        }
        entity db "Database" {
            type: database
            level: container
            description: "Stores user registration, auth, account data"
        }

        web -> app "JSON/HTTPS"
        app -> db  "JDBC"
    }

    customer -> web "Visits pages using HTTPS"
    customer -> app "Uses API using HTTPS"
    banking -> email "Sends e-mail using SMTP"
}

将 view: context 时，投影保留 customer、banking（折叠 group）、email，并聚合组内关系为 customer → banking、banking → email。

8. DSL 与 AST 规范

8.1 与现有规范的关系

  • 图表类型仍为 diagram architecture（见 architecture.md）。
  • level、view、scope 注册为标准属性（与 type 同级），进入 attributes.standard。
  • 不新增箭头类型；C4 的「同步/异步」用 relation 标签或 meta.protocol 表达。

8.2 校验规则（validation 增量）

规则 ID级别规则
C4-001Errorview 值必须在枚举内
C4-002Errorlevel 值必须在枚举内
C4-003Warning声明 view 但无 entity/group 带 level
C4-004WarningComponent 实体不在任何 level:container group 内
C4-005WarningContext 视图出现 level:component 实体（投影前检查）
C4-006Errorscope 指向不存在的 id
C4-007WarningSoftware System group 嵌套超过 2 层（与全局 group 限制一致）
C4-008Warning粗视图专用 relation（两端均在将被折叠的子图内）在细层视图中不可见 — 提示改写在细层

8.3 diff2 扩展

level、view、scope 变更纳入 ChangeTarget::Diagram | Entity | Group 属性 diff。 PR 审查文案示例：「Container api 下新增 Component PaymentService」。

9. 视图投影算法

函数签名建议：project_c4_view(diagram: &mut Diagram) -> Result<(), C4ProjectError>

9.1 步骤（确定性顺序）

投影在副本上执行，不修改 canonical model（见 §17、§18）。

  1. 克隆 diagram → projected_diagram（保留原 AST 供 diff / 多视图）。
  2. 读取 view；若缺失则 return（no-op）。
  3. 收集 带 level 的 entity / group，按 id 排序。
  4. 过滤 保留 level_order(level) ≤ level_order(view) 的元素（landscape 仅 context 系统间关系）。
  5. 折叠 group：若 view 粗于组内子级别，将 group 提升为单一超级节点（保留 label、level、description）。
  6. 关系聚合：按 §18.2 规则合并跨折叠边；多边标签默认 ; 拼接。
  7. scope 裁剪（component 视图）：仅保留 scope 指定 group 及其子 entity、相关边。
  8. 注入 drilldown：为可下钻节点写入 drilldown 元数据（导出 HTML/SVG 时用）。
  9. 清理 悬空 relation；按 relation 源目标 id 排序。

9.2 关系聚合示意

9.3 API 层多视图批量渲染

POST /render

12. Agent / MCP 指引

12.1 图种选择

用户意图输出
「C4 上下文 / 系统全景 / 谁用谁」architecture + view: context
「C4 容器 / 微服务 / 技术栈」architecture + view: container
「模块内部 / Controller Service Repo」architecture + view: component + scope
「登录时序 / OAuth 步骤」sequence（非 C4 dynamic）
「业务流程 / 审批步骤」flowchart

12.2 System Prompt 片段（建议写入 MCP Skill）

绘制 C4 图时使用 diagram architecture，并遵守：
1. 必须设置 view: context | container | component 之一
2. 每个 entity/group 必须设置 level，与 view 层级体系一致
3. Software System 用 group 包裹其 Container
4. Person 用 type:frontend + level:context；外部系统用 type:external
5. 关系标签写清协议（HTTPS / JSON / JDBC / Kafka）
6. 不要使用 Mermaid 的 C4Context / C4Container 关键字

12.3 常见错误与修复提示

错误修复建议
view:container 但全是 level:context为 Web/API/DB 添加 level:container
Component 散落在顶层放入 level:container 的 group 内
混淆 flowchart步骤顺序场景改用 flowchart

17. 四层组织与多视图架构

关键心智模型：C4 的「四层」不是一张图里的四个图层（像 Photoshop），而是同一套模型上的四个缩放级别——类似地图从全国到街区。每一级通常渲染为一张独立视图，而不是四层叠在一起。

17.1 存储 vs 展示

17.2 组织方式对比

方式做法评价
推荐 单模型 + view 投影 一个 .dfy，渲染时指定 view 或 CLI --c4-views Diff、Agent、ID 一致性最好；与 Structurizr 模型/视图分离同构
推荐 单模型 + 批量导出 CI 一次导出 context.svg + container.svg + manifest.json 文档站 / PR / 合规归档最实用
推荐 HTML 多视图包 单个 .html 内嵌多视图 + 侧栏切换 + 点击下钻 评审演示体验最佳；P2 交付
避免 四个独立 .dfy context.dfy / container.dfy 各写各的 entity id 漂移、关系不一致、diff 无意义
例外 企业多域拆分 Compose 按业务域生成多 AST，用 meta.source_id 对齐 仅规模化场景；Landscape 再合并，非手写四文件

17.3 在 DSL 中如何「长在一起」

用 group 表边界 + level 表缩放级，全写进同一 diagram architecture：

  • Software System → 顶层 group，level: context
  • Container → group 内 entity（或「含 Component 的 Container」本身也是 group），level: container
  • Component → Container group 内 entity，level: component
  • Person / External → 顶层 entity，level: context

渲染某一级时：设置 diagram 的 view，引擎投影后只 layout 该视图可见子图。不需要为每一层各维护一套 relation 副本。

概念数量说明

逻辑视图3–5context / container / component×N / deployment（可选）
导出成品N 张 SVG/PNG + 1 manifest或 1 个 HTML 包内含全部
Code 层图0默认不做；类图交给 IDE

17.5 综合架构（端到端）

存储：     1 × canonical model（.dfy / JSON AST）
投影：     N × view（context | container | component[+scope]）
层内关系： relation 写在最细层 + 粗层由聚合推导（§18）
层间导航： drilldown 元数据 → HTML / SVG 超链接（§18.3）
默认导出： SVG×N + manifest.json [+ PNG]
企业归档： ZIP（dfy + svg + png + manifest + diff 报告 + snapshot-id）
精修通道： draw.io（单视图导出，.dfy 仍为 source of truth）
动态交互： diagram sequence（另图；entity id 与模型对齐）

18. 链接语义：relation、包含与下钻

C4 里存在三种不同的「链接」，必须用不同机制表达，不能混为一种箭头语法。

18.1 三种链接对照

类型含义Drawify 表达示例
A. 业务关系 调用、数据流、依赖 relation（-> / --> / <->） api -> db "JDBC"
B. 包含关系 系统包含容器、容器包含组件 group 嵌套 + group_id（不是 relation） group banking { entity api ... }
C. 视图导航 从 Context 图点进 Container 详图 drilldown 元数据（不是 relation） 点击 banking → 打开 container 视图
常见错误：用 banking -> api 表示「系统包含 API」——包含应靠 group；banking -> email 才是跨系统业务关系。

18.2 业务关系：写在细层，粗层靠聚合

原则：relation 写在模型中最细、最准确的一层（通常为 container 或 component）。粗视图禁止再手写一套「context 专用边」。

聚合规则（投影时确定性执行）

规则 ID条件动作
AGG-01 边两端都被折叠进同一 group 删除该边（内部流量，粗视图不可见）
AGG-02 边一端在 group 外，另一端在 group 内 将组内端点提升为 group 节点：customer -> web ⇒ customer -> banking
AGG-03 边两端在不同 group 两端分别提升到各自 group（若已折叠）
AGG-04 投影后存在多条相同 (from, to) 边 合并为一条；标签用 ; 连接（可配置为取首条）
AGG-05 自环在粗视图 丢弃或标 warning（架构图自环通常无意义）

聚合示例

// 模型（细层 relations）
customer -> web "HTTPS"
web -> app "JSON"
app -> db "JDBC"
banking -> email "SMTP"    // banking 为 group id

// view: context 投影后
customer -> banking "HTTPS"           // AGG-02：web 被折叠
banking -> email "SMTP"               // 跨 group 边保留

18.3 视图导航：drilldown 元数据

Context 图上点击「Internet Banking」应打开 Container 视图——这是超链接，不是图上的箭头。

AST / Scene 字段（P1）

// entity 或 group 的 standard 属性，或 ExportNode 扩展字段
"drilldown": {
  "view": "container",
  "scope": "banking",       // 可选；component 视图时必填
  "title": "Internet Banking — Containers"
}

自动生成规则

当前 view节点生成的 drilldown
context level:context 的 group（Software System） { view: "container", scope: <group_id> }
container 含 component 子节点的 group 或 container entity { view: "component", scope: <id> }
component — 无下钻（或 P3 链到外部 Code 工具 URL）

各导出格式中的下钻能力

格式下钻实现优先级
HTML 多视图包 JS 切换 view + scope；侧栏列视图列表 P2 体验最佳
SVG <a xlink:href="container.svg#banking"> 包裹节点 P1
draw.io 链接到另一 Page（单视图导出后人工可改） P2
PNG 无交互；文档内并排多图 + 标题导航 静态兜底

18.4 隐含关系（Implied Relationships）

P2 Structurizr 可根据层级自动推导「父到子」的隐含边。Drawify 初期不实现，要求作者写显式 relation；待聚合稳定后再评估，避免 Agent 困惑。

18.5 Dynamic View 与 sequence 的关系

C4 Dynamic View（按时间展示交互）不塞进 architecture 投影。 使用 diagram sequence，participant id 与模型 entity id 对齐； manifest 中可增加 dynamic_views 条目指向对应 sequence 文件或内嵌 DSL 片段。

19.1 格式优先级（实用导向）

梯队格式主要用户典型用途

19.3 CLI 多视图导出

# 从单一模型批量导出默认 C4 视图集
drawify export model.dfy \
  --c4-views context,container \
  --format svg \
  --output-dir ./dist/views/

# 同时生成 PNG 与 manifest
drawify export model.dfy \
  --c4-views context,container,component:app \
  --format svg,png \
  --c4-manifest ./dist/manifest.json

# 合规 ZIP 包（dfy + 全部视图 + manifest + 可选 diff）
drawify export model.dfy \
  --c4-bundle zip \
  --c4-views context,container \
  --snapshot-id banking-2026-06-25 \
  --output ./dist/banking-architecture.zip

# 交互式 HTML 包（P2）
drawify export model.dfy \
  --c4-bundle html \
  --c4-views context,container,component:app \
  --output ./dist/banking-c4.html
