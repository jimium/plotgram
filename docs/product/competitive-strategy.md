# Drawify 竞争策略：三个方向的取舍分析

> 版本：0.1.0-draft | 状态：战略分析

## 核心结论

Drawify 要在与 Mermaid 的竞争中建立壁垒，必须在**语义微调**方向 All in，而非图形美观或布局合理。

| 维度 | 能否取胜 | 原因 | 建议投入 |
|------|----------|------|----------|
| 图形美观程度 | ❌ 不能 | 主观、无止境、Mermaid 追平成本低、不构成壁垒 | 做到"不丑"即可 |
| 布局合理程度 | ⚠️ 必要但不充分 | 是及格线不是决胜点，Graphviz 布局好但没赢 | 做到 80 分，确保不拖后腿 |
| 语义微调支持 | ✅ 唯一胜算 | Mermaid 架构天花板、AI 时代刚需、企业场景杀手锏 | **All in**，这是核心壁垒 |

**一句话：Drawify 要赢 Mermaid，不是靠"画得更好看"或"排得更整齐"，而是靠"改得更精准"。**

---

## 1. Mermaid 的护城河在哪

Mermaid 的核心壁垒不是技术，而是**生态**：

- GitHub 原生渲染、Notion 集成、VSCode 插件
- 75k Stars、无数教程与社区贡献
- "谁都在用"的网络效应

在"谁都能用、谁都在用"这个维度上，Drawify 不可能打败 Mermaid。

因此，竞争策略的核心问题是：**在哪一个维度上，Mermaid 的设计假设恰好成了它的天花板？**

---

## 2. 图形美观程度 — 不能取胜

### 2.1 为什么不能取胜

- **美观是主观的、无止境的**。今天比 Mermaid 好看一点，明天 Mermaid 社区一个 PR 就追平了。这是红海竞争。
- **Mermaid 的渲染质量已经够用了**。对绝大多数用户来说，Mermaid 的 SVG 输出已经"够好看"，不是痛点。
- **美观不构成护城河**。Excalidraw 证明了"手绘风"可以差异化，但那是一个全新品类，不是在同一个赛道上比谁更精致。
- **投入产出比极低**。从"还行"到"惊艳"需要海量设计工作（字体、配色、阴影、动画），但用户迁移成本远大于审美提升带来的收益。

### 2.2 建议策略

做到"不丑、专业、不丢人"即可。具体：

- 预置主题覆盖主流场景（已有 20+ 主题 JSON）
- 默认主题保持清晰专业，不追求视觉惊艳
- 不在渲染细节上与 Mermaid 竞赛

---

## 3. 布局合理程度 — 必要但不充分

### 3.1 为什么是必要条件

- **Mermaid 的布局确实是痛点**。Sugiyama 算法在复杂图上交叉严重、长边回溯、子图嵌套后布局崩坏，这是公认问题。
- **布局是"及格线"能力**。布局差到影响阅读时，用户会直接放弃；但布局好到一定程度后，边际收益急剧下降。没有人因为"布局再好 5%"就换工具。
- **Graphviz 的布局比 Mermaid 好得多，但 Graphviz 并没有因此取代 Mermaid**。这本身就证明了布局好 ≠ 赢。

### 3.2 布局的间接价值

布局有一个关键的间接价值：**它是"语义微调"的物理基础**。

没有好的布局，微调就无从谈起：

- Layout Intent Refinement 需要一个合理的 base layout 作为起点
- "把认证服务放右边"的前提是系统已经有一个可读的初始布局
- Diff 高亮渲染需要稳定的布局 seed，避免"同一张图两次渲染位置完全不同"

所以布局要做到 **80 分以上**，确保不拖后腿，但不需要追求 95 分。

### 3.3 建议策略

- Sugiyama / 力导向等核心算法做到稳定可用
- Layout Intent Overlay 作为差异化能力重点投入（但价值归属"语义微调"维度）
- 布局 seed 保证同一 AST 渲染结果可复现
- 不在"比 Mermaid 少两个交叉"上消耗过多资源

---

## 4. 语义微调支持 — 唯一胜算

### 4.1 Mermaid 的设计假设在这里制造了天花板

Mermaid 的设计假设是 **"人类写一次，渲染一次"**。它没有考虑过：

- 写完之后还需要**迭代修改**（Agent 生成 → 用户反馈 → 局部调整）
- 两个版本之间需要**语义级比较**（不是文本 diff）
- 错误需要**结构化反馈**来驱动自动修复

这些不是 Mermaid "做得不好"的问题，而是 Mermaid **从架构上就不可能做好**的问题：

- AST 不是一等公民，无法做语义级操作
- 没有结构化错误模型，错误时渲染空白
- 没有语义 Diff/Patch，只能文本替换

要加这些能力，等于重写 Mermaid。

### 4.2 语义微调恰好是 AI 时代的刚需

AI 生成图表的典型工作流是：

```
Agent 生成初版 → 用户说"把认证服务放右边" → Agent 局部调整 → 渲染
```

Mermaid 在这个流程中断裂在第二步：

- 用户反馈 → Agent 只能**重新生成整段文本**（出错率飙升）
- 或者 Agent 做**文本级 diff**（毫无语义意义）

Drawify 的语义微调能力恰好解决了这个断裂：

| 能力 | 解决的问题 | Mermaid 能否做到 |
|------|-----------|-----------------|
| AST Patch | 只改一个属性，不动其他 | ❌ |
| Layout Intent Overlay | 只调一个位置，不重新布局 | ❌ |
| 结构化错误 + fix | Agent 一次重试就能修复 | ❌ |
| 语义 Diff | "新增了哪个实体，修改了哪个属性" | ❌ |
| Diff 高亮渲染 | 新增绿、删除红、修改黄 | ❌ |

### 4.3 企业场景的杀手级需求也在这里

| 企业需求 | Mermaid | Drawify |
|----------|---------|---------|
| 架构变更 Diff | 不支持 | 语义 Diff + 高亮渲染 |
| 增量 Patch | 不支持 | AST Patch 事务性修改 |
| 变更可追踪 | 不支持 | snapshot-id + source_id |
| 合规归档 | 截图 | PNG 企业参数 + 溯源元数据 |
| PR 门禁集成 | 无意义文本 diff | 语义 diff + markdown 报告 |

这些能力全部建立在"语义微调"的基础上。没有 AST 一等公民 + Diff/Patch，企业场景一个都打不进去。

### 4.4 语义微调的能力拆解

"语义微调"不是单一功能，而是一组相互支撑的能力：

```
                    ┌─────────────────────┐
                    │   语义微调（核心壁垒）  │
                    └──────────┬──────────┘
           ┌──────────────────┼──────────────────┐
           ▼                  ▼                  ▼
    ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
    │  结构化错误    │  │  AST Diff /   │  │  布局意图     │
    │  + 修复建议   │  │  Patch        │  │  约束层       │
    └──────┬───────┘  └──────┬───────┘  └──────┬───────┘
           │                  │                  │
           ▼                  ▼                  ▼
    Agent 自我修正      版本对比与增量修改    局部位置微调
    1 次重试修复        企业变更可追踪       不推翻全局布局
```

三者缺一不可：

- **没有结构化错误**：Agent 生成失败后无法精准修复，退化为 Mermaid 的"重试大法"
- **没有 AST Diff/Patch**：企业场景无法做变更对比，退化为截图对比
- **没有布局意图约束**：微调只能推翻重布局，退化为"改一个字，整图乱跳"

### 4.5 语义动画：让"精准"变得可感知

SVG 动画本身不是壁垒，但**语义动画**是"语义微调"策略的视觉放大器——它让 Drawify 的核心优势从"技术上更精准"变成"体验上可感知"。

#### 装饰性动画 vs 语义动画

| 类型 | 示例 | 战略价值 | 建议 |
|------|------|----------|------|
| 装饰性动画 | 节点弹跳出现、悬停发光、连线粒子 | 低，属于"图形美观"维度 | 不做 |
| **语义动画** | Diff 过渡、Patch 可视化、布局意图反馈 | **高，强化核心壁垒** | **P1 投入** |

装饰性动画归入"图形美观"维度，Mermaid 社区补齐成本低，不构成壁垒。语义动画则完全不同——它服务于"变更可感知"这个独特需求。

#### 最有杀伤力的场景

用户说："把认证服务从网关左边移到右边"

- **Mermaid**：重新生成整段文本 → 重新渲染 → 用户看到一张全新的图，需要自己对比哪里变了
- **Drawify 无动画**：Patch 生效 → 新图替换旧图 → 同样需要对比
- **Drawify 有语义动画**：Patch 生效 → `auth` 节点平滑滑动到新位置 → 用户**零成本**感知变更

这个差异在**连续交互**场景下被放大：Agent 迭代 5 次微调，每次变更都是平滑过渡 vs 每次整图刷新。

#### 语义动画的具体类型

| 动画场景 | 战略归属 | 价值 |
|----------|----------|------|
| **Diff 过渡动画**：旧图 → 新图的平滑变形 | 语义微调 | 让"改得更精准"肉眼可见 |
| **Patch 应用动画**：高亮被修改的节点/边 | 语义微调 | 用户一眼看到改了什么 |
| **布局意图动画**：节点从旧位置滑到新位置 | 语义微调 | Layout Intent 的可视化反馈 |
| **数据流动画**：沿边方向流动的光点 | 语义表达 | 表达"谁调谁"，比箭头更直觉 |

#### 技术实现思路

SVG 原生支持 `<animate>` / `<animateTransform>`，不需要 JS 运行时：

- 节点位置变更 → `<animateTransform type="translate">`
- 节点新增 → `<animate attributeName="opacity">` 0→1
- 节点删除 → `<animate attributeName="opacity">` 1→0
- 边路径变更 → `<animate attributeName="d">`

关键设计点：动画数据由渲染器在 Patch/Diff 时自动生成，用户和 Agent 不需要控制动画参数。

#### 优先级

| 优先级 | 动画类型 | 理由 |
|--------|----------|------|
| **P1** | Diff/Patch 过渡动画 | 直接强化核心壁垒，企业 Architecture Compare 的体验差异化 |
| **P1** | 布局意图过渡动画 | Layout Intent Overlay 的可视化反馈，让微调可感知 |
| **P2** | 数据流方向动画 | 语义增强，对架构图/时序图有实际价值 |
| **不做** | 纯装饰动画 | 投入产出比低，属于"图形美观"维度 |

### 4.6 建议策略

**All in 语义微调**，具体优先级：

1. **P0 — 结构化错误全链路**：Parser → Validator → WASM → Server → CLI 全部输出结构化 JSON 错误 + fix payload
2. **P0 — Diff 补全 + source_id 对齐**：属性 modify、关系变更、group 变更；跨快照实体对齐
3. **P0 — Diff 高亮渲染模式**：新增绿、删除红、修改黄，配合亚秒级 SVG 重绘
4. **P0 — AST Patch 事务性**：增量修改不重新生成，失败回滚
5. **P1 — Layout Intent Overlay**：right_of / near / pin 等约束，base layout 之上最小扰动
6. **P1 — AST → DSL round-trip**：Compose 产出 AST 后可序列化为可读 .dfy

## 5. 竞争策略总结

```
          高 ┌─────────────────────────────────────┐
             │                                     │
   竞 争 壁 垒 │            ★ 语义微调支持             │
             │         （Mermaid 架构天花板）          │
             │                                     │
          中 │     ┌───────────────┐               │
             │     │  布局合理程度   │               │
             │     │ （必要条件）    │               │
             │     └───────────────┘               │
             │                                     │
          低 │ ┌───────────┐                       │
             │ │ 图形美观   │                       │
             │ │ （及格即可）│                       │
             │ └───────────┘                       │
             └─────────────────────────────────────┘
               低            中            高
                      Mermaid 追平难度
```

- **图形美观**：Mermaid 追平难度低，不构成壁垒 → 及格即可
- **布局合理**：Mermaid 追平难度中，是必要条件 → 做到 80 分
- **语义微调**：Mermaid 追平难度高（需架构重写），是唯一壁垒 → All in

## 7. drawify.studio 场景下的策略修正

### 7.1 新场景改变了美观的投入标准

前面的分析基于一个前提：**Drawify 的第一用户是 AI Agent**，人类是间接消费者。在这个前提下，"好看"不是核心诉求。

但 drawify.studio 引入了一个新场景：**人类直接使用，输出进入 PPT/Word**。用户链路变为：

```
AI Agent 场景：Agent 生成 → 机器渲染 → 人类阅读（关注"对不对"）
studio 场景：  人类编辑 → 机器渲染 → 人类展示（关注"好不好看"）
```

当图进入 PPT/Word 时，它不再是"技术文档里的辅助说明"，而是**展示材料的一部分**。用户会拿它和 Canva、Figma 产出的图放在一起比较——这时候"不丑"就不够了。

### 7.2 美观的投入标准提升

| 层次 | 含义 | AI Agent 场景 | drawify.studio 场景 |
|------|------|--------------|-------------------|
| 不丑 | 清晰、专业、不丢人 | 必须做到 | 必须做到 |
| **好看** | 精致、有设计感、能上 PPT | 不需要 | **必须做到** |
| 惊艳 | 独特视觉风格、品牌辨识度 | 不需要 | 差异化机会 |

**在 drawify.studio 场景下，"好看"从"锦上添花"变成了"及格线"。**

### 7.3 现有竞品的空白

| 工具 | 视觉质量 | 能否上 PPT | 短板 |
|------|----------|-----------|------|
| Mermaid | 及格 | 勉强 | 风格单一，无法定制 |
| Draw.io | 及格偏上 | 可以 | 偏工具感，不够精致 |
| Lucidchart | 较好 | 可以 | 商业产品，风格偏企业 |
| Excalidraw | 独特 | 特定场景 | 手绘风不适合正式场合 |
| PlantUML | 差 | 不行 | 丑是公认的 |
| **空白** | **精致 + 语义驱动** | **可以** | **没有人做到** |

**没有一款文本/语义驱动的出图工具，能产出"PPT 级"的图。** 这是一个真实的空白。

### 7.4 与"语义微调"不矛盾

容易犯的错误：认为"要好看"就等于"要放弃语义微调去拼美观"。

实际上，drawify.studio 的核心竞争力仍然是**语义驱动**，只是最终交付物的质量标准提高了：

```
drawify.studio 的价值公式 = 语义驱动编辑 × PPT 级输出质量
                            ↑                    ↑
                         核心壁垒（Mermaid 做不到）  新要求（Mermaid 做不好）
```

- 如果只有"语义驱动"但输出丑 → 用户会回到 Draw.io 手动画
- 如果只有"好看"但没有语义驱动 → 用户会回到 Canva/Figma
- **两者结合** → "用语义的方式，产出 PPT 级的图" → 这是独占位置

### 7.5 具体策略

| 能力 | 说明 | 优先级 |
|------|------|--------|
| **导出质量** | 高 DPI PNG、透明背景、无水印 | P0 |
| **主题系统深化** | 从 20+ 主题扩展到"场景化主题包"（商务汇报、技术分享、学术论文） | P1 |
| **品牌色 / 企业色注入** | 用户指定品牌色，主题自动适配 | P1 |
| **字体与排版** | 中文字体优化、行间距、对齐 | P1 |
| **节点样式库** | 精心设计的图标/形状库，而非简单几何体 | P2 |
| **PPT 插件** | 直接在 PPT 中插入 Drawify 图，双击可编辑 | P2 |

### 7.6 修正后的竞争策略矩阵

```
                    AI Agent 场景          drawify.studio 场景
图形美观     及格即可               需要做到"好看"（PPT 级）
布局合理     80 分                  80 分（同）
语义微调     All in（核心壁垒）       All in（核心壁垒，不变）
```

**语义微调的核心壁垒地位不变**，但图形美观的投入标准从"不丑"提升到"能上 PPT"。这不是战略转向，而是**交付标准的提升**。

两个场景共用同一套 Core 引擎，差异仅在渲染层的主题/样式/导出质量上。Core 团队继续聚焦语义微调，studio 团队负责在渲染输出侧补齐"PPT 级"体验。
