Drawify Core Pipeline

DSL 源码 → 最终输出的完整调用链路  ·  涉及 mod / 入口函数一目了然

Input
DSL 源码（.dfy 文本）
用户编写的 Drawify DSL，例如 diagram flowchart { entity a "A" ... }
1
Parse
pipeline::parse → dsl::parser
dsl::lexer::tokenize()
词法分析：源文本 → Token 流（关键字、标识符、箭头符号等）
dsl::parser::parse()
语法分析：Token 流 → Diagram AST，识别 entities / relations / groups / attributes / style_decls
dsl::parser::parse_with_diagnostics()
降级解析入口：即使有错误也尽量返回部分 AST（fallback diagram），同时收集结构化 errors / warnings
中间产物
RawDiagram
Parser 直接产出（封装 Diagram）；可能缺少 entity.type 等默认属性，style 未物化，layout_plan 未解析
2
Prepare
pipeline::prepare → prepare::*
prepare::apply_profile_defaults()
按图表类型 profile 补全缺失的 entity.type（如 flowchart 默认 process，state 默认 state）
prepare::resolve_style_context()
解析主题：StyleRequest { theme_id, dark_mode } + diagram theme 属性 → ThemeIdResolver → builtin_style_sheet → ResolvedStyleContext
prepare::validate_style_decls()
校验 style 声明块（错误阻断，警告收集）
prepare::materialize_styles()
将 theme cascade 物化到每个 entity / relation 的 attributes.style（fill, stroke 等），内联 style 不被覆盖
PreparedDiagram::new()
封装 diagram 并调用 LayoutPlan::resolve：解析 layout_algo / edge_routing 及其 option，挂在 layout_plan() 上供下游复用
debug_assert_prepared_invariants()
[debug only] 断言 PreparedDiagram 不变量 I1（style_decls 已清空）/ I2（entity.type 已补全）/ I3（style 含 fill 或 stroke）
核心中间产物
PreparedDiagram
满足不变量的规范图表：所有 entity 有 type，style 已物化，layout_plan 已解析，可被下游直接消费
3
Validation
validation::validate → common / attrs / kinds / icons / layout
common::validate_diagram_attributes()
校验 diagram 级属性：layout / layout_algo / edge_routing / theme / graphic_style / group_sizing / snap / title 等
attrs::validate_entity_attributes()
校验 entity 标准属性键（profile::STANDARD_ENTITY_ATTRS）+ style 属性值域；TYPE 值域按 profile 开放或闭集
icons::validate_entity_semantic_icon()
校验 semantic / icon 属性：未知 semantic 发 W301，未知 icon 发 W302，icon: none 不警告新增
attrs::validate_relation_attributes()
校验 relation 标准属性键（profile::STANDARD_RELATION_ATTRS）+ style 属性
common::validate_relations()
检查 from/to 引用是否存在，group 不可直接参与连线
common::validate_groups()
校验 group 属性（style/color/layout），空 group 发出警告
common::check_orphan_entities()
检测无任何连线的孤立实体（warning 级别）
kinds::validate_diagram_type()
图表类型专属规则分派：flowchart / state / mindmap / sequence / architecture / er（经 kinds::registry::entry_for 分派）
layout::algorithm_config::validate_algorithm_config_warnings()
布局配置块的语义校验警告（算法名 / option 是否合法）
layout::validate_layout_plan_warnings()
基于已解析 LayoutPlan 的警告（option 组合是否合理）新增
校验产出
ValidationResult { errors, warnings }
有 error 则中断（CLI exit 1 / Server 返回错误 / WASM 不进入渲染）；只有 warning 或为空则继续
4
Render（layout → scene → encode）
pipeline::render → scene + encode
scene::compute_layout()
入口：PreparedDiagram → LayoutResult（复用 layout_plan，避免重复解析）
分支：若 format == Ascii，跳过 build_scene 视觉物化，直接走 ascii::encode_direct(diagram, layout, options) → RenderOutput::Text。 ASCII 只消费 diagram + layout，不需要 fill/stroke/width 等视觉属性。
⚙ layout::compute_layout_with_plan — 布局计算（嵌入 scene::compute_layout）
validate_layout_config()
校验布局配置：算法是否存在、是否适用于当前图表类型；layout 方向 / edge_routing 合法性
registry::build_layout_strategy(algo, plan).compute()
执行具体布局策略：计算节点坐标 + 分组包围框（plan.layout_algo + plan.layout_options）
grid_snap::snap_layout_to_grid()
[可选] 网格吸附：节点对齐到像素网格，刷新分组与画布边界
registry::build_edge_routing_strategy().route()
边路由：为每条边计算几何路径（path 点 + 控制点 + 标签位置 + from/to port）
refine::run_refine()
layout ↔ route 反馈循环：检测折线穿障 → 局部推开问题节点 → re-route新增
grid_snap::snap_edge_waypoints()
[可选] 边路径点网格吸附
布局算法注册表（LAYOUT_ALGORITHM_NAMES）

边路由算法注册表（EDGE_ROUTING_NAMES）

✦ scene::build_scene — 视觉物化（ASCII 跳过此步）
request.resolve_context()
解析主题 ID → builtin_style_sheet / explicit_style_json → resolve_style_sheet → ResolvedRenderContext（含 ResolvedStyleContext + GraphicStyleId + icon_resolve）
kinds::entry_for(diagram_type)
从图表类型注册表获取 DiagramTypeEntry（函数指针：materialize_node_style / materialize_edge_style / paint_export_node / paint_export_edge / paint_svg_defs）
entry.materialize_node_style()
为每个节点物化最终基础视觉属性（NodeStyle：shape / fill / stroke / font 等）
entry.materialize_edge_style()
为每条边物化最终基础视觉属性（EdgeStyle：stroke / arrow / label 等）
color_queries::canvas_background() / title_color() / group_fill() / group_stroke() / group_label_color() / group_style_by_depth()
查询画布背景色、标题色、分组填充/描边/标签色（按 depth 派生颜色，z_index 排序确保嵌套层级正确）
渲染器无关的中间场景
ExportScene { prepared, canvas, layout, context, nodes[], edges[], groups[] }
已解析主题 + 已计算布局 + 已物化视觉属性，供所有格式编码器共享（ASCII 不经过此层）
5
Format Encode
render::encode::encoder_for(format).encode_scene()
render::encode::encoder_for(format)
按 RenderFormat 选择具体 FormatEncoder（静态单例：SVG_RENDERER / ASCII_RENDERER / JSON_RENDERER / PNG_RENDERER / WEBP_RENDERER）
SVG
render::encode::svg
SvgRenderer.encode_scene()
→ paint::scene_svg::encode
→ entry.paint_export_node/edge
→ paint::svg_utils
ASCII
render::encode::ascii
AsciiRenderer
→ encode_direct（独立路径）
跳过 build_scene
只用 diagram + layout
JSON
render::encode::json
JsonRenderer
→ serde 序列化
ExportScene
PNG
render::encode::png
PngRenderer
→ rasterize SVG
[feature: raster]
需 fonts_dir
WebP
render::encode::webp
WebpRenderer
→ rasterize SVG
[feature: raster]
需 fonts_dir
Final Output
RenderOutput::Text(String)  |  RenderOutput::Binary(Vec<u8>)
SVG / ASCII / JSON → Text   |   PNG / WebP → Binary

pipeline 模块入口函数

编排逻辑集中在 pipeline/，只做串联，不实现具体逻辑。各服务模块提供纯函数。
pipeline::parse(source)
→ Result<RawDiagram>
仅解析，不 prepare / validate
pipeline::prepare(raw, style_request)
→ Result<PrepareOutput>
规范化 RawDiagram → PreparedDiagram（含 layout_plan 解析）
pipeline::parse_prepare(source, style_request)
→ PipelineOutput
解析 + prepare，跳过 validate（diff / export / patch 场景）
pipeline::parse_prepare_validate(source, style_request)
→ PipelineOutput
解析 + prepare + validate。CLI / Server / WASM 渲染与校验的标准入口
pipeline::run(source, style_request, format)
→ PipelineResult<RenderOutput>
端到端：DSL 源码一步到 RenderOutput
pipeline::render_output(request)
→ Result<RenderOutput>
渲染编排：compute_layout → (ASCII 分支 | build_scene → encode)
pipeline::render_text(request)
→ Result<String>
渲染为文本（SVG / ASCII / JSON），二进制格式报错
pipeline::render_bytes(request)
→ Result<Vec<u8>>
渲染为字节（PNG / WebP），文本格式报错
pipeline::render_json(diagram)
→ String
便捷入口：从 PreparedDiagram 直接渲染 JSON（diff / export 场景）
pipeline::render_with_style_json(diagram, format, style_json)
→ Result<RenderOutput>
使用内联 style JSON 渲染（CLI / Server 传入自定义主题）

三个前端入口的调用序列

CLI
renderparse_prepare_validate() → RenderRequest::new() + render_text() / render_bytes()，错误时 exit(1)
validateparse_prepare_validate()，错误时 exit(1)
exportdiffpatch使用 parse_prepare()（跳过 validate）；diff 调 diff::diff()，export 调 render_json()
Server
api.rs::check_source() → pipeline::parse_prepare_validate()
渲染端点：校验通过后 render_output(&RenderRequest)；HTTP 响应头携带 X-Drawify-Valid / X-Drawify-Format / X-Drawify-Warnings
WASM
render(source, format) / render_with_options(source, format, options_json) → parse_prepare_validate() → 无 error 则 render_output(&request)（单格式）
另有 parse_to_json()（仅 parse）与 validate() 独立入口；产出 JSON：{ success, format, text, errors, warnings }
