🧠 Mindmap · 思维导图布局

中心辐射 / 单向树布局，支持 radial / top-to-bottom / left-to-right 方向，根 / 主干 / 分支 / 叶节点差异化尺寸
O(V+E) 确定性 10–200 nodes mindmap.rs

核心思想

以根节点为中心，子节点沿指定方向（径向 / 自上而下 / 自左向右）逐层展开。 不同层级节点采用差异化尺寸：根节点最大，主干次之，分支较小，叶节点最小—— 视觉上自然呈现"主干→分支→叶子"的层次感。

方向模式

radial · 中心辐射（默认）
top-to-bottom · 自上而下树
left-to-right · 自左向右树

节点尺寸分级

层级相对尺寸说明
Root（根）最大居中放置，视觉焦点
Main（主干）较大一级子节点，粗线条连接
Branch（分支）中等二级及以下子节点
Leaf（叶）最小无子节点的末端

适用场景

  • 知识树：概念层级、分类体系
  • 会议笔记：主题→要点→细节
  • 学习路径：课程→章节→知识点

✅ 优势

  • 天然符合"中心→外围"认知模型
  • 差异化尺寸让层级一目了然
  • 支持三种方向，灵活适配场景
  • 确定性输出，O(V+E) 线性复杂度

⚠️ 劣势

  • 仅支持树结构，非树图需先提取生成树
  • 大宽度树在 radial 模式下外围节点可能过密
  • 不支持交叉边 / 回边

🔄 Circular · 环形布局

节点沿圆环排列，边沿弧线路由，环排序 + 弧边路由一体化
O(V+E+R) 确定性 10–120 nodes circular.rs

核心思想

将所有节点均匀分布在圆周上，边使用弧线路由（arc routing）而非直线连接。 环上的节点排序经过优化，使频繁交互的节点相邻，减少弧线交叉。 R 代表弧线路由的额外计算量。

关键特性

  • 环排序优化：根据邻接关系调整圆周顺序，减少边交叉
  • 弧线边路由：边沿圆弧绘制，避免直线穿越圆心区域
  • 闭环友好：天然适合有环 / 闭环关系的图

适用场景

  • 状态机：状态之间的转移形成闭环
  • 闭环关系图：循环依赖、反馈回路
  • 环形流程：迭代流程、周期性过程

✅ 优势

  • 闭环关系一目了然
  • 弧线边避免视觉混乱
  • 所有节点地位平等（无"顶层"偏见）
  • 确定性输出

⚠️ 劣势

  • 节点超过 120 个时圆周过密
  • 不适合有明显层次 / 方向性的图
  • 弧线路由增加渲染复杂度
  • 非连通图需额外处理

🏗️ Architecture V2 · 架构图专用布局

两层 Sugiyama：宏观层分组拓扑分层 + 微观层组内节点分层，分组感知紧凑排列
O(V+E+G×S) 确定性 20–250 nodes architecture_v2.rs

核心思想

专为架构图设计的混合布局算法，结合 Sugiyama 分层布局的层级优化能力和分组感知的紧凑排列能力。 核心思路是两层 Sugiyama：宏观层将分组视为"超级节点"做拓扑分层，确定组的层级位置； 微观层对组内节点做 Sugiyama 分层，同时考虑跨组边的端口位置。 坐标分配采用 Brandes-Kopf 风格，确保组边界紧凑。

关键特性

  • 分组感知分层：组作为整体参与拓扑排序，同组节点保持紧凑
  • 跨组边端口：微观层布局考虑跨组边的连接方向
  • 重叠消除：后处理阶段检测并消除节点/分组重叠
  • 宽高比优化：调整分组间距使整体布局更均衡

核心常量（来自 architecture_v2.rs）

LAYER_GAP = 80 px
NODE_GAP = 48 px
GROUP_GAP_X = 50 px
CROSSING_SWEEPS = 12
GROUP_PADDING = 28 px

适用场景

  • 架构图：微服务拓扑、系统组件图
  • 分组组件图：需要体现模块边界的依赖图
  • 有分组语义的有向图：层次 + 分组双重约束

✅ 优势

  • 分组感知，同组节点紧凑排列
  • 确定性输出，适合自动化流水线
  • 比力导向更清晰的层级结构
  • 比纯 Sugiyama 更好的分组语义

⚠️ 劣势

  • 仅适用于 DiagramType::Architecture
  • 分组间拓扑复杂时宏观层分层可能不够紧凑
  • 两层 Sugiyama 计算量较大
  • 不支持无分组语义的图

📨 Sequence · 序列图布局

参与者水平展开，消息按时间步纵向排列，布局直接生成消息路径（跳过通用边路由）
O(V+E) 确定性 10–200 msgs sequence.rs

核心思想

序列图布局与传统图布局有本质区别：参与者（participant）水平排列， 消息（message）按时间步纵向排列。布局引擎直接生成消息的箭头路径， 完全跳过通用边路由阶段，因为序列图的消息路径是确定的（水平箭头 + 可选的自身回调折线）。

关键特性

  • 参与者生命线：每个参与者一条垂直虚线
  • 消息路径直出：布局引擎直接计算箭头起止坐标
  • 自身消息：支持参与者发给自己（折线路径）
  • 同步 / 异步：实线箭头 = 同步调用，虚线箭头 = 异步返回

适用场景

  • API 交互：客户端→服务端→数据库的请求/响应流
  • 服务调用链：微服务之间的调用时序
  • 协议序列：通信协议的握手 / 数据交换步骤

✅ 优势

  • 时序语义清晰，消息顺序一目了然
  • 跳过通用边路由，性能极佳
  • 确定性输出
  • 天然支持"请求→响应"的往返表达

⚠️ 劣势

  • 仅适用于序列图语义，非通用布局
  • 参与者过多时水平方向拥挤
  • 不支持分支 / 并行 / 循环结构
  • 消息过多时纵向过长

🧲 Force-Directed · 分组感知力导向布局

Fruchterman-Reingold 力导向 + 组内引力 / 组间斥力，适合架构图与流程图
O(I·V²) 非确定性 10–180 nodes force_directed.rs

核心思想

基于 Fruchterman-Reingold 力导向：节点间库仑斥力、边弹簧引力，通过模拟退火迭代收敛。 在 FR 基础上增强分组语义——同组节点互相吸引形成簇，不同组之间保持分离。 多连通分量独立布局后并排打包。

适用场景

  • 架构图：组件关系复杂、需自然散开，同时保留 group 边界
  • 流程图：网状依赖、层次不明显的流程
  • 微服务拓扑：服务间多向依赖、无明显阅读顺序

✅ 优势

  • 真实力导向，边长分布自然
  • 分组引力保留模块语义
  • 适合网状、弱层级图

⚠️ 劣势

  • 非完全确定性（迭代 + 初值）
  • 大图 O(V²) 成本较高
  • 有明确阅读顺序时不如 Sugiyama / architecture-v2

✨ Sugiyama V2 · 增强分层布局

贪心环消除 + NS 式 rank 压缩 + BK 式 4-pass 中位对齐 + 紧凑化
O(V+E+S×C) 确定性 20–320 nodes sugiyama_v2.rs

核心思想

Sugiyama V2 是 V1 的增强版，在四个阶段都做了升级： 贪心环消除替代 DFS 反转， NS 式 rank 压缩让层更紧凑， BK 式 4-pass 中位对齐让水平分布更均匀， 最后再做紧凑化（compaction）消除多余空白。 S = sweep 次数，C = 每层节点数。

V1 vs V2 对比

阶段Sugiyama V1Sugiyama V2
环消除 DFS + back-edge 反转 贪心环消除（更少反转边）
层分配 最长路径 NS 式 rank 压缩（更紧凑的层间距）
交叉最小化 Barycenter + up/down sweeps BK 式 4-pass 中位对齐（更均匀的水平分布）
坐标分配 迭代 barycenter + 网格对齐 4-pass 对齐 + 紧凑化 compaction
节点容量 ~150 20–320

适用场景

  • 高质量流程图：需要更少交叉、更紧凑的布局
  • 复杂 DAG：节点多、分支 / 汇合密集
  • 需要均匀水平分布：BK 4-pass 对齐让同层节点间距更一致

✅ 优势

  • 比 V1 更紧凑、更均匀
  • 贪心环消除反转边更少
  • NS rank 压缩减少"空层"
  • BK 4-pass 对齐水平分布更优
  • 支持更大图（320 节点）

⚠️ 劣势

  • 比 V1 计算量更大（4-pass + compaction）
  • 实现复杂度高，调试困难
  • 对简单图可能"过度优化"
  • 紧凑化可能让边过于密集
