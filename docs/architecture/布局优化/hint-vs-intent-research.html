Drawify · 布局调整方案研究

Hint 还是 Intent?
面向 Agent 的布局调整路线抉择

基于 drawify 现有实现、792 样本量化数据、行业竞品与 2025 学术趋势的综合研判。 报告回答两个问题:哪条路线更值得投入?Hint 方案的扩展与能力提升是否可行?

研究范围 · 实现 + 竞品 + 学术 数据基线 · Phase 0–3, 792 样本 日期 · 2026-06-20
核心结论

Intent 更值得投入;Hint 退守为"自动质量基线层",两者协同而非互斥

drawify 的 DSL 是面向 Agent 的。Agent(LLM)擅长表达定性的关系约束("A 应在 B 下方"、"三者对齐"), 而不擅长产出数值化的密度/坐标 hint。Intent 的语义模型与 Agent 的自然语言输出天然对齐; Hint(friendliness)的价值在于无需任何人参与的自动兜底,应作为 Intent 之下的默认质量地板保留。 扩展 Hint 的表达力会不可避免地向 Intent 收敛——投入产出比递减。

1背景:面向 Agent 的 DSL 与布局调整的两条路

drawify 的 DSL 设计以 Agent 为一等公民——图由 LLM 生成文本、文本解析为 AST、AST 驱动布局。 当自动布局"基本可读但局部不满意"时,如何让 Agent(或人)做局部修正?项目探索过两条路:

Hint 路线(提示)

声明式、轻量、算法解读。用户在 DSL 写 layout: fan-out,或算法自行产出 FriendlinessReport 诊断并自动调整。Agent/用户不直接控制具体节点对,只给"方向性提示", 由算法自由解读。代表实现:GroupLayoutHint、FriendlinessAdjuster。

Intent 路线(意图)

命令式、显式、逐对约束。Agent/用户通过 overlay 声明 below(a,b)、align_vertical([a,b,c]), 算法原生消费并返回每条意图的满足状态(Satisfied/Partial/Conflicted)。 代表实现:LayoutIntentOverlay + RefinementReport。

关键事实: 两条路线在 compute_layout_with_plan_and_overlay 调度层已经协同共存 (见 layout/mod.rs)。Intent 的 PinSet 会被 FriendlinessAdjuster 尊重—— 这不是"二选一",而是"谁主谁辅"的投入分配问题。

Intent 路线的结构性优势:
  • 可观测:每条意图都有明确状态,Agent 能据此迭代("这条 Conflicted 了,我换种说法")
  • 可组合:拓扑意图改 rank,几何意图改坐标,正交不冲突
  • 不破坏契约:overlay 独立透传,diagram 不变异,relations[i]↔edges[i] 索引保持
  • 语义对齐 Agent:below/above/align 是自然语言概念,LLM 几乎零成本产出

4行业与竞品分析

布局调整能力是 text-to-diagram 工具的核心差异化点。下表汇总主流竞品的布局控制模型。

工具 方向控制 相对位置/对齐 绝对坐标 约束(rank/weight) 路线归类
Graphviz / DOT rankdir group, rank=same pos constraint, weight Intent 为主(约束驱动)
D2 (TALA) direction near(对象) top/left(TALA) 有限 Hint + Intent 混合
D2 (dagre/elk) direction near(仅常量) — — Hint 为主
Mermaid TB/LR/RL/BT — — — 纯 Hint(极简)
PlantUML direction together — — Hint 为主

Graphviz 的启示

作为最强大的图布局工具,Graphviz 的控制力几乎全部来自约束型属性(rank=same、 constraint=false、weight、group、pos)—— 这些本质都是 Intent(用户显式声明约束,算法满足)。pos 绝对坐标是兜底,不是主力。 drawify 的 Intent 设计与 Graphviz 同源,但更克制(拒绝绝对坐标,只做轴约束)。

D2 的分化

D2 自研的 TALA 引擎支持 top/left 绝对锁定和 near 对象相对定位—— 这是向 Intent/显式控制靠拢的信号。而 dagre/elk 后端只能用 direction 这种 Hint。 D2 的演进方向说明:能力上限由 Intent 决定,Hint 只是入门门槛。

2025 学术趋势:Agent 生成图,定性关系约束胜过数值

See it. Say it. Sorted.(NeurIPS 2025)

Sketch-to-diagram 的 Agent 系统,核心是 Critic-Candidates-Judge 循环。 Critic VLM 提出"一小批定性的、关系性的编辑"(如"A 应在 B 右侧"), 而非数值坐标。论文明确写道:"prioritizes qualitative reasoning over brittle numerical estimates, preserves global constraints (e.g., alignment, connectivity)"。 这正是 Intent 的语义模型。

arXiv:2508.15222

Co-Layout: LLM-driven(arXiv 2511.12474)

用 LLM 从文本生成 scene graph(含约束),再做优化。LLM 产出的是"房间与家具列表 + 关联约束", 不是坐标。结论:LLM 适合产出结构化约束,数值优化交给求解器。 分工 = Intent(LLM 产出) + 算法(求解),与 drawify 的 Intent overlay 架构一致。

arXiv:2511.12474
行业共识收敛: 能力强的工具(Graphviz/TALA)靠 Intent/约束 拉开上限; 2025 年 Agent 生成图的前沿研究也选择定性关系约束而非数值 hint。 drawify 作为面向 Agent 的 DSL,与这条主线天然对齐。

5决策:为什么 Intent 更值得投入

综合实现成熟度、Agent 语义对齐、行业趋势、可观测性四个维度。

决策维度HintIntent胜出
Agent 语义对齐 差——Agent 不会产出密度/坐标 优——below/align 是自然语言 Intent
可观测性/可迭代 差——无逐条状态 优——RefinementReport 逐条反馈 Intent
局部稳定(不跳变) 差——改 DSL 整图重算 优——overlay 局部修正 Intent
零成本兜底 优——无需任何人参与 差——必须显式声明 Hint
行业能力上限 低——Mermaid/PlantUML 级 高——Graphviz/TALA 级 Intent
实现成熟度 高——Phase 0-3 已验收 中——核心已实现,缺 benchmark Hint(暂)
扩展表达力天花板 低——加类型即收敛到 Intent 高——right_of/same_rank/near 可演进 Intent
最终判断: 7 个维度中 Intent 胜 5、Hint 胜 2。Hint 的两个胜项(零成本兜底、当前成熟度)都是防御性价值—— 保留即可,不值得继续扩张。Intent 的胜项全是进攻性价值——决定 drawify 能走多远。 建议:Intent 主攻,Hint 守底。

6Hint 方案扩展与能力提升的可行性

用户特别要求研究 Hint 的扩展空间。结论:可行但天花板低,且越扩展越像 Intent。

6.1 三条扩展路径与可行性评估

路径 A · 新增 DSL Hint 类型

可行性:低。加 density/spread 这类 hint,本质是给算法一个"软偏好"。 但 Phase 3 已证明:算法内部参数(barycenter 权重)对扰动极度敏感,任何非默认值都让交叉增加。 且这类 hint 一旦要"精确控制具体节点对",就必然演化成 below/align——收敛到 Intent。

路径 B · Adjuster 增强

可行性:中。当前 adjuster 只优化 predicted_crossings。 可扩展为多目标(加 edge_length、port_conflict),或用学习到的权重替代分族静态权重。 但 Phase 2 已暴露本质代价:用边长换交叉(edge_length +1.5%)。多目标会陷入帕累托权衡,收益递减。

路径 C · V3 目标集成

可行性:低。Phase 3 实测:Sugiyama barycenter 权重方案反效果已禁用; force-directed RUDY 排斥力有效但仅限力导向族。把友好度塞进布局目标函数,会破坏算法的确定性, 且与 Intent 的显式约束优先级冲突(谁说了算?)。

6.2 Hint 扩展的根本矛盾

收敛定理: Hint 的表达力每提升一档,就更像 Intent 一分。
  • layout: fan-out(现状) → 算法自由解读,纯 Hint
  • align: a,b,c(假设扩展) → 这就是 align_vertical Intent
  • order: a before b(假设扩展) → 这就是 below/above Intent
一旦 Hint 需要"指名道姓"控制具体节点,它就不再是 Hint,而是 Intent。 所以扩展 Hint = 重新发明 Intent,且没有 RefinementReport 的可观测性。

6.3 Hint 真正值得做的有限增强

Hint 不应追求"表达力",而应巩固"自动兜底"这个唯一护城河。建议只做 3 件事:

  • 补齐 Intent 的 benchmark:用现有 792 样本框架,量化 Intent(below/align)对 enc/跳变的影响——这是 Intent 的短板,但用 Hint 的评测基础设施来补
  • Adjuster 尊重 PinSet:当前 adjuster.rs L13 已预留 pinned 接口但未实装,补完它让 Intent+Hint 协同更稳
  • FriendlinessReport 暴露给 Agent:把热点信息结构化进 RefinementReport,让 Agent 能读"哪里不友好"并主动下 Intent——Hint 成为 Intent 的传感器
结论: Hint 的扩展不应是"加更多 hint 类型",而是"让 hint 更好地服务 intent"。 把 FriendlinessReport 从"给人看的诊断"升级为"给 Agent 下的意图建议",是投入产出比最高的方向。

中期 · Intent 扩展

right_of / same_rank(P2)

远期 · DSL 化

arrange 语法(高频意图进 AST)

把高频 Intent(below/align)提升为 DSL 一等语法,Agent 可直接在文本里写,无需 overlay API。这是 Intent 的终态。

Intent 投入分配(主攻)

  • 70% · 扩展意图类型 + DSL arrange 语法
  • 20% · 跨算法支持(力导向/放射类原生消费)
  • 10% · 效果 benchmark 与回归

Hint 投入分配(守底)

  • 60% · FriendlinessReport 结构化反哺 Intent
  • 30% · Adjuster pinned 协同 + 回归
  • 10% · 维护现有五维度量(不新增类型)
