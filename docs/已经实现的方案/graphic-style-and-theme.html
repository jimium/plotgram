Graphic Style 与 Theme 架构

Drawify 将配色与排版 token（theme）和几何笔触皮肤（graphic style）拆成两条正交维度， 使同一套色板可以叠加不同画法，而无需为每种组合重写布局或图表逻辑。

一句话总结

theme 回答「画成什么颜色、字号、背景」； graphic_style 回答「节点和边在 SVG 里怎么画出来」。

两者在 ResolvedRenderContext 中合并，渲染时先读取 theme 物化后的 fill / stroke，再交给 GraphicStylePainter 决定描边参数、填充纹理、滤镜与箭头形态。

GraphicStylePainter 抽象

graphic_style 模块的核心是 GraphicStylePainter trait（策略模式）。 每个风格（standard、excalidraw、blueprint…）实现该 trait，并在 painter_for() 注册。

兜底机制

NodeShape::render_with_context 先调用 graphic_painter().render_node_shape()。 若返回 None（如 standard 风格），则回退到默认几何 SVG（<rect>、<path> 等）。 因此新增风格可以只覆盖需要差异化的形状，而不必重写全部图表渲染器。

解析优先级

GraphicStyleId 枚举定义在 spec/profile.rs（规格层），而非 graphic_style 模块内。 这样 DiagramProfile.default_graphic_style 与 layout、edge_routing 等默认值同级， 且 spec 不必依赖 SVG 绘制实现，避免循环依赖。

模块边界（刻意不交叉）

  • layout — 坐标与边路径，与视觉皮肤无关
  • theme — StyleSheet cascade，物化颜色/字号 token
  • kinds — 图表语义、实体规则、scene 物化与绘制分派
  • graphic_style — 笔触策略与 SVG 画法，读取 NodeStyle 中的色值
  • render::encode — 格式编码（SVG / PNG / WebP / ASCII / JSON）

graphic_style 不参与 prepare 阶段的 AST 改写； theme 不决定手绘抖动或点描算法。 二者仅在 ResolvedRenderContext 汇合。
