1架构总览

Drawify Studio 采用分层架构,从用户输入到图表渲染经过 5 层处理,每层职责单一、单向流动:

UI 层 ChatPanel / PreviewCanvas / DslViewer · 用户交互与结果展示
↓
Hook 层 useAgent / useWasm · 状态管理与副作用编排
↓
Agent 层 AgentLoop / tools / prompt / context · LLM 对话循环与 Tool 调用
↓
WASM 桥接层 lib/wasm.ts · TypeScript ↔ Rust 类型转换
↓
drawify-core parse → prepare → validate → layout → scene → encode

核心设计原则

  • LLM 驱动:用户用自然语言描述需求,LLM 生成 Drawify DSL 并通过 Tool 调用渲染
  • Agent 绘图:LLM 通过 function-calling 操控 6 个 Tool(render/validate/parse/diff/apply_patch/layout_catalog)
  • 单格式输出:每次 render 调用只产出一种格式(默认 SVG),符合 drawify-core 的单格式渲染约定
  • DSL 即真相:context.source 是图表的唯一真实状态,SVG 是其渲染产物

2Agent 核心循环

AgentLoop 是 Agent 的核心引擎,实现 THINKING → TOOL_EXEC → EVALUATE → RESPOND 循环,支持多轮 Tool 调用与错误自修复。

用户输入
"画一个微服务架构图"
↓
buildMessages()
system prompt + 当前DSL + 历史 + 输入
↓
循环 (最多 10 次)
LLM.chat(messages, tools)
↓
有 tool_calls?
↓
否
↓
返回最终回复
AgentResult { message, source, svg, diff }
是
↓
执行 Tool
render / validate / parse / diff / apply_patch
↓
更新 context
source 同步 / 记录 lastSvg / lastDiff
↓

继续下一轮迭代

循环退出条件

  • 正常退出:LLM 返回无 tool_calls 的回复,进入 RESPOND 阶段
  • 迭代上限:达到 maxIterations=10,返回"请尝试更明确的指令"
  • 错误退出:LLM 调用失败(网络/鉴权/限流),返回错误消息
关键设计:Tool 执行失败不会中断循环,而是将错误信息作为 tool result 返回给 LLM,让 LLM 有机会自修复(如修正 DSL 语法错误后重新 render)。

3.2 DSL 生成与 source 同步

LLM 通过 render tool 的 source 参数传入生成的 Drawify DSL。这里有一个关键的 source 同步机制:

// src/agent/tools.ts - render executor
render: async (args, context) => {
  const wasm = await getWasm();
  const source = String(args.source ?? '');
  const format = String(args.format ?? 'svg') as RenderFormat;
  const options = args.options as RenderOptions | undefined;
  const optionsJson = options ? JSON.stringify(options) : undefined;

  // ★ LLM 通过 render 传入新生成的 DSL,同步到 context 以便 DslViewer 展示
  if (source && source !== context.source) {
    context.source = source;
  }

  return renderSource(wasm, source, format, optionsJson);
},
为什么在 render tool 中同步 source?
P0 阶段(单轮生成)LLM 只调用 render 生成图表,不调用 apply_patch。如果不在 render 时同步 source,DslViewer 将看不到 LLM 生成的 DSL。P1 阶段(增量编辑)则由 apply_patch 更新 source。

3.3 WASM 渲染管线

renderSource 是 TypeScript 与 Rust WASM 的桥接函数,负责调用 WASM 并解析返回的 JSON:

// src/lib/wasm.ts
export function renderSource(
  wasm: DrawifyWasm,
  source: string,
  format: RenderFormat,
  optionsJson?: string,
): RenderResult {
  const json =
    optionsJson && typeof wasm.render_with_options === 'function'
      ? wasm.render_with_options(source, format, optionsJson)
      : wasm.render(source, format);

  return safeParse<RenderResult>(json, {
    success: false,
    format,
    text: null,
    errors: ['无法解析渲染结果'],
    warnings: [],
  });
}

WASM 侧(Rust)的渲染管线遵循 drawify-core 的分层架构:

// crates/drawify-wasm/src/lib.rs - render_impl
fn render_impl(source, format_str, options) -> String {
    // 1. 解析格式
    let format = RenderFormat::from_str(format_str)?;

    // 2. parse → prepare → validate
    let output = parse_prepare_validate(source, &style_request);

    // 3. 构造 RenderRequest
    let mut request = RenderRequest::new(&prepared, format);

    // 4. 应用 options(theme/graphic_style/dark_mode)
    apply_options_to_request(&mut request, options.as_ref())?;

    // 5. layout → scene → encode
    match render_output(&request) {
        Ok(RenderOutput::Text(text)) => {
            // 返回 { success:true, text:"<svg>...</svg>" }
        }
        Ok(RenderOutput::Binary(_)) => { /* png/webp */ }
        Err(e) => { /* 错误诊断 */ }
    }
}

3.4 SVG 提取与 UI 更新

AgentLoop 在每次 render tool 执行后,检查结果并记录 lastSvg:

// src/agent/AgentLoop.ts
// 记录最后一次渲染的 SVG
if (toolCall.name === 'render' && result && typeof result === 'object') {
  const renderResult = result as { success: boolean; text?: string };
  if (renderResult.success && renderResult.text) {
    lastSvg = renderResult.text;  // ★ 提取 SVG 字符串
  }
}

循环结束后,useAgent 从 result 中更新 React state:

// src/hooks/useAgent.ts
const result = await runAgentLoop(text, contextRef.current, config);

// AgentLoop 会原地修改 contextRef.current.source(通过 render/apply_patch tool)
if (contextRef.current.source && contextRef.current.source !== currentSource) {
  setCurrentSource(contextRef.current.source);  // DSL 更新
}
if (result.svg) {
  setCurrentSvg(result.svg);  // SVG 更新
}
最终效果:PreviewCanvas 实时显示渲染的 SVG,DslViewer 展示 DSL 源码,ChatMessage 用 Markdown 渲染 Agent 的自然语言回复。

3.5 tool_calls 格式转换(关键修复)

内部 ToolCall 是扁平结构,但 DeepSeek/OpenAI API 要求嵌套结构。buildRequestBody 负责转换:

内部格式(TS)API 格式(发送给 LLM)
{ id, name, arguments: object } { id, type: "function", function: { name, arguments: "JSON字符串" } } 
// src/lib/llm.ts - buildRequestBody
const apiMessages = messages.map((msg) => {
  if (msg.role === 'assistant' && msg.tool_calls?.length) {
    return {
      role: 'assistant',
      content: msg.content || null,  // 空字符串转 null
      tool_calls: msg.tool_calls.map((tc) => ({
        id: tc.id,
        type: 'function',  // ★ 必须有
        function: {
          name: tc.name,
          arguments: typeof tc.arguments === 'string'
            ? tc.arguments
            : JSON.stringify(tc.arguments ?? {}),  // ★ 必须是字符串
        },
      })),
    };
  }
  return msg;
});

4Tool-Calling 机制

Agent 通过 6 个 Tool 操控 drawify-wasm,每个 Tool 对应一种能力:

Tool 用途 WASM 绑定 阶段
render 渲染 DSL 为 SVG/ASCII/JSON wasm.render / render_with_options P0
validate 校验 DSL 语法语义,返回错误诊断 wasm.validate P0
parse 解析 DSL 为 AST JSON wasm.parse_to_json P0
layout_catalog 查询可用布局算法 wasm.layout_catalog P0
diff 比较两份 DSL 的语义差异 wasm.diff_sources P1 待实现 P1
apply_patch 对 DSL 应用增量补丁(Change 列表) wasm.apply_patch P1 待实现 P1

Tool 执行器签名

export type ToolExecutor = (
  args: Record<string, unknown>,
  context: AgentContext,

5上下文管理

AgentContext 是 Agent 的状态容器,贯穿整个对话生命周期:

export interface AgentContext {
  source: string;              // 当前生效的 DSL 源码(图表真相)
  diagramType: DiagramKind | null;  // 图表类型
  history: ChatMessage[];       // 对话历史
  maxIterations: number;       // 最大迭代次数(默认 10)
}

buildMessages 的上下文注入策略

每次 runAgentLoop 调用时,buildMessages 重新组装发送给 LLM 的消息列表:

1
System Prompt
固定的 SYSTEM_PROMPT,描述 Agent 能力、DSL 语法、工作流程
2
当前 DSL
如果 context.source 非空,注入为 system message,让 LLM 知道当前图表内容
3
对话历史
最近 10 条 user/agent 消息(仅 content,不含 tool_calls)
4
当前用户输入
本次用户消息
// src/agent/prompt.ts
export function buildMessages(userMessage, context): LLMMessage[] {
  const messages = [
    { role: 'system', content: SYSTEM_PROMPT },
  ];

  // 注入当前 DSL 状态
  if (context.source) {
    messages.push({
      role: 'system',
      content: `当前图表的 DSL 源码如下,后续修改基于此版本:\n\n\`\`\`drawify\n${context.source}\n\`\`\``,
    });
  }

  // 注入对话历史(最近 10 条)
  const recentHistory = context.history.slice(-10);
  for (const msg of recentHistory) {
    if (msg.role === 'user') messages.push({ role: 'user', content: msg.content });
    else if (msg.role === 'agent') messages.push({ role: 'assistant', content: msg.content });
  }

  // 当前用户输入
  messages.push({ role: 'user', content: userMessage });

  return messages;
}

历史压缩

compactHistory 在每轮对话后调用,保留最近 20 条消息,避免上下文无限增长:

export function compactHistory(context, keepLast = 20): AgentContext {
  if (context.history.length <= keepLast) return context;
  return { ...context, history: context.history.slice(-keepLast) };
}
上下文处理要点:Tool 调用历史(tool_calls + tool results)不持久化到 context.history,只在单次 runAgentLoop 的 messages 数组中存活。这意味着跨轮次的 Tool 调用上下文会丢失,但 LLM 能通过注入的"当前 DSL" system message 知道图表状态,足以支持迭代修改。

响应格式统一

不同 Provider 的响应格式不同,parseResponse 负责统一为 LLMResponse:

function parseResponse(provider, data): LLMResponse {
  if (provider === 'anthropic') {
    return parseAnthropicResponse(data);  // content blocks: text | tool_use
  }
  return parseOpenAIResponse(data);  // choices[0].message.tool_calls
}

DeepSeek/Ollama/Custom 都走 OpenAI 兼容格式,Anthropic 走原生 messages 接口。
